# 第六章：应用层

## 6.1 网络应用模型

### 6.1.1 客户/服务器模型（C/S）
- 特征：客户端与服务器地位不对称，服务器处于中心位置，客户进程之间不直接通信。
- 优点：便于集中管理。
- 缺点：服务器资源受损将影响全局；中心服务器负载大，其性能决定了整个系统的性能。

### 6.1.2 P2P 模型
- 特征：去中心化，主机地位平等，主机之间直接进行通信。
- 优点：单个节点损坏不影响全局；各个节点可分摊负载。
- 缺点：主机在享受服务的同时也要为其他主机提供服务，自身负载大；可能导致大量网络数据流和网络拥塞。

## 6.2 域名系统 DNS

### 6.2.1 概述
- 域名：给互联网上的主机、路由器起的具有层次化特性的名字。
- DNS 作用：实现域名与 IP 地址的相互转换，方便人类记忆与机器处理。
- 基本特点：
  - 属于应用层协议，通常基于 **UDP**，使用端口 **53**。
  - 采用 C/S 模式。
  - 分布式系统，由大量域名服务器层次化组织。

### 6.2.2 域名结构
- 层次化命名：`…… 三级域 . 二级域 . 顶级域`
- 标号由英文、数字和连字符 `-` 组成，不区分大小写。
- 每个标号 ≤63 字符，完整域名（含点）≤255 字符。

### 6.2.3 顶级域分类
- 国家/地区顶级域（ccTLD）：如 `.cn`, `.us`
- 通用顶级域（gTLD）：如 `.com`, `.net`, `.org`
- 基础结构域名：`.arpa`，用于反向域名解析

### 6.2.4 域名服务器
- **根域名服务器**：最高层次的域名服务器，全球共 13 组（任播）。
- **顶级域名服务器**：负责管理本级顶级域名下属的所有二级域名。
- **权限域名服务器**：负责一个具体区域的域名管理，能将其管辖的主机名转换为 IP 地址。
- **本地域名服务器**：通常位于用户 ISP 内，当主机发出 DNS 查询请求时，查询报文首先送至该服务器。

### 6.2.5 域名解析过程
- **递归查询**：接收查询的域名服务器若不知道所需 IP 地址，会代替请求者继续向其他域名服务器查询，直到得到结果，然后返回给请求者，期间请求者只发出一次请求。
- **迭代查询**：接收查询的域名服务器若不知道，则向请求者返回另一个可能知道的域名服务器的 IP 地址，请求者再向该服务器发起新的查询。

  典型解析流程（逐级迭代）：
  1. 主机向本地域名服务器发出递归查询请求。
  2. 本地域名服务器依次向根域名服务器 → 顶级域名服务器 → 权限域名服务器进行迭代查询。
  3. 得到结果后返回给主机。
  若本地域名服务器缓存了对应 IP 地址，则直接返回。

### 6.2.6 补充速记（考试/复习要点）
- **域名层级方向**：域名层次自右向左递增，最右为根（`.`，通常省略），最左为主机名（例如 `www.example.com.`）。
- **服务器四级划分**：根 → 顶级域（TLD）→ 权威域名服务器 → 本地域名服务器（递归/缓存服务器，用户默认 DNS），本地域名服务器不是层级顶端且负责缓存与递归查询。
- **传输层承载**：常用 UDP/53；当响应超过 512 字节或进行区域传送时使用 TCP/53（区域传送必须用 TCP）。
- **域名长度限制**：单个标号 ≤ 63 字符，完整域名（含点）≤ 255 字符。
- **缓存要点**：本地域名服务器会缓存结果以减轻上级查询，但缓存受 TTL 限制且可能导致短时间内解析不一致。

  主机也可请求迭代查询，但通常默认使用递归。

## 6.3 文件传输协议 FTP

### 6.3.1 功能与特点
- 支持不同操作系统间的文件传输。
- 支持按用户权限进行文件管理。
- 支持匿名 FTP，共享公共文件。
- 采用 C/S 模式，基于 **TCP**。
- 控制信息与数据分开传送（**带外传输**）。
- 文件修改效率低（需整体下载后再上传）。

### 6.3.2 工作方式
- **控制连接**：用于传输命令和应答信息，贯穿整个 FTP 会话期间，保持不断开；客户端随机端口 → 服务器 **21** 端口。
- **数据连接**：传输文件或文件列表时动态建立，传输完毕后立即释放，属于非持久连接。

### 6.3.3 传输模式
- **主动模式（PORT）**：客户端通过控制连接告知服务器某个随机端口号，服务器使用 **20** 端口主动连接客户端该端口。
- **被动模式（PASV）**：客户端发送 PASV 命令，服务器开放一个随机端口并告知客户端，客户端向该端口发起连接。

### 6.3.4 服务器进程
- 主进程：只有一个，监听 **21** 端口，负责接收新请求。
- 从属进程：每个用户登录后对应一个控制进程，在整个 FTP 会话期间存在。
- 数据传送进程：为每次数据传输临时创建和销毁。

## 6.4 电子邮件

### 6.4.1 系统组成
- **用户代理（UA）**：如邮箱客户端（Foxmail、Outlook）或 Web 邮箱界面。
- **邮件服务器**：存放用户邮件，同时担任发送和接收的角色（SMTP 客户端/服务器双重身份）。
- **电子邮件协议**：**SMTP** 用于发送，**POP3** 或 **IMAP4** 用于接收。

### 6.4.2 邮件格式与 MIME
- **基本格式**：邮件 = 信封 + 内容；内容 = 首部 + 主体；首部必须包含 `To`、`From`，`Subject` 可选。
- **MIME（多用途互联网邮件扩展）**  
  - 原因：SMTP 只能传输 7 位 ASCII 编码的纯英文文本，无法直接支持中文、多媒体等。  
  - MIME 扩展：
    - 新增多个首部字段（如 `MIME-Version`, `Content-Type`, `Content-Transfer-Encoding`）。
    - 定义了多种主体格式和传送编码，能将任意数据编码为 7 位 ASCII 码。

### 6.4.3 SMTP 协议
- 作用：将邮件从发送方用户代理推向发送方邮件服务器，以及从发送方邮件服务器推至接收方邮件服务器。
- 服务器与服务器之间也通过 SMTP 进行推送（push）。
- 特点：C/S 模式，基于 **TCP**，服务器端口 **25**。
- 三个阶段：连接建立 → 邮件传送 → 连接释放。

### 6.4.4 POP3 协议
- 作用：接收方用户代理从邮件服务器拉取邮件。
- 特点：C/S 模式，基于 **TCP**，服务器端口 **110**。
- 两种工作方式：
  - 下载并删除：将邮件下载到本地后删除服务器副本。
  - 下载并保留：下载后仍保留服务器副本。

### 6.4.5 IMAP4 协议（补充）
- **IMAP（Internet Message Access Protocol）**：用于从邮件服务器读取与管理邮件。
  - 明文端口：**143**
  - 加密端口（IMAPS/TLS）：**993**
- **核心特点（对比 POP3）**
  - 邮件保留在服务器，本地以缓存/视图为主，多设备状态一致。
  - 支持文件夹/标签管理，支持已读/未读/星标/删除等状态同步。
  - 支持按需获取邮件头、正文或附件，节省带宽。
- **协议形态**：IMAP 是行式文本协议，命令与响应均为纯 ASCII 文本。

#### 工作流程（简化）
1. 客户端建立 TCP 连接，服务器返回欢迎语。
2. 客户端登录并选中文件夹（如 `INBOX`）。
3. 查询邮件、拉取内容、标记状态、删除邮件。
4. 发送 `LOGOUT` 结束会话。

#### 常用命令（编程常见）
命令格式：`标签 命令 [参数]`（标签用于匹配请求/响应）。

| 命令 | 作用 |
|:---|:---|
| `LOGIN user pass` | 登录 |
| `LIST "" *` | 列出所有文件夹 |
| `SELECT INBOX` | 选中文件夹 |
| `FETCH n (BODY[TEXT])` | 获取正文 |
| `FETCH n (FLAGS)` | 查看状态 |
| `STORE n +FLAGS (\Seen)` | 标记已读 |
| `STORE n +FLAGS (\Deleted)` | 标记删除 |
| `EXPUNGE` | 删除已标记邮件 |
| `LOGOUT` | 退出 |

#### 状态标志（FLAGS）
- `\Seen`：已读
- `\Answered`：已回复
- `\Flagged`：星标/重要
- `\Deleted`：标记删除
- `\Draft`：草稿

#### 版本与安全
- 主流使用 **IMAP4rev1**（RFC 3501）。
- 生产环境必须启用 TLS（IMAPS/993），明文 143 易被抓包。
- 现代邮箱常用 OAuth2 认证，减少明文密码登录。

#### 常见应用场景
- 多端设备同步的邮件客户端
- 企业告警/工单系统拉取邮件

## 6.5 万维网 WWW

### 6.5.1 万维网组成
- **统一资源定位符（URL）**：`<协议>://<主机>:<端口>/<路径>`  
  端口和路径有时可省略（如 HTTP 默认端口 80，默认首页 index.html）。
- **超文本传输协议（HTTP）**：应用层协议，基于 TCP，定义了请求报文与响应报文的格式和交互规则。
- **超文本标记语言（HTML）**：描述页面内容与格式的标记语言。

### 6.5.2 HTTP 工作方式
浏览器访问一个网页的过程：  
1. 首先获取 HTML 文件；  
2. 解析 HTML 后获取内嵌的其他资源（图片、CSS、JS 等），每个资源都需要一次独立的 HTTP 请求/响应。

- **非持续连接**：每次 TCP 连接只传送一个 HTTP 请求报文和一个响应报文，传送后立即释放。
  - 可并行建立多个 TCP 连接以加速资源获取。
- **持续连接**：一次 TCP 连接可以传送多个 HTTP 请求和响应。
  - 非流水线方式：上一个请求收到响应后才能发送下一个请求。
  - 流水线方式：可以连续发送多个请求，而不必等待响应。

  HTTP/1.0 默认采用非持续连接；HTTP/1.1 默认采用持续连接（并可选择流水线）。

| 维度 | HTTP 1.0 | HTTP 1.1 |
|:---|:---|:---|
| 默认连接模式 | 默认非持久（每对象单独握手） | 默认持久 keep-alive |
| 是否支持流水线 | 否 | 是（同一 TCP 上可连续发多个请求） |
| Host 头部 | 可选 | 必选（同 IP 多虚拟主机靠它区分） |
| 状态码与方法集 | 基础（200 / 301 / 400 / 404 / 500 等少数） | 扩充（含 100 / 206 / 409 / 503 等更多）+ 新增 OPTIONS / PUT / DELETE / TRACE / CONNECT 等方法被标准化 |
| 默认底层协议 / 端口 | TCP 80 | TCP 80 |

> **注意**：即便是 HTTP/1.1 的流水线模式，也需要先获取到 HTML 文件后，才能解析出其中引用的其他资源（如图片、CSS、JS 等）的 URL，然后再向服务器发起请求。因此流水线并非一次请求所有资源，而是对同一连接内的请求进行排队优化。

### 6.5.3 HTTP 报文格式
#### 请求报文
```
请求行（方法 URL 版本 CRLF）
首部行（字段名: 值 CRLF）
空行（CRLF）
实体主体（通常用于 POST 请求）
```
- 常用方法：
  - **GET**：请求读取由 URL 标记的信息。
  - **HEAD**：与 GET 类似，但只读取首部，不返回实体主体。
  - **POST**：向服务器提交数据（如表单）。
  - **PUT**：上传文件到指定 URL。
  - **DELETE**：删除文件。
  - **CONNECT**：建立隧道，常用于代理。
  - **OPTIONS**：查询服务器支持的方法。

#### 响应报文
```
状态行（版本 状态码 短语 CRLF）
首部行
空行
实体主体（通常为请求的资源内容）
```
- 常见状态码：
  - `1xx`：信息提示。
  - `2xx`：成功（`200 OK`）。
  - `3xx`：重定向（`301 Moved Permanently` 永久移动，`302 Found` 临时移动，`304 Not Modified` 使用缓存）。
  - `4xx`：客户端错误（`400 Bad Request`，`404 Not Found`）。
  - `5xx`：服务器错误（`500 Internal Server Error`）。

### 6.5.4 Cookie
- 作用：用于在无状态的 HTTP 中维持状态信息，实现个性化服务。
- 工作过程：  
  1. 客户端首次访问服务器，服务器在响应中添加 `Set-Cookie` 首部，返回 Cookie。  
  2. 客户端后续访问时，在请求中附加 `Cookie` 首部，服务器以此识别用户。  
- Cookie 具有一定的安全风险，不宜存储敏感信息。

**Cookie 与 Session 对比**

| 维度 | Cookie | Session |
|:---|:---|:---|
| 存储位置 | 客户端浏览器 | 服务器端 |
| 安全性 | 较低（客户端可见，易被窃取/伪造） | 较高（数据存于服务器） |
| 数据大小 | 单个 Cookie ≤ 4 KB，每域一般 ≤ 20 个 | 受服务器内存限制，理论无上限 |
| 生命周期 | 可设置过期时间，可长期保存 | 通常随浏览器关闭或超时后失效 |
| 服务器压力 | 不占用服务器存储 | 占用服务器内存 |
| 典型配合 | 服务器通过 Set-Cookie 下发 Session ID，客户端后续请求携带该 ID，服务器据此找到对应 Session | — |

### 6.5.5 Web 缓存 / 代理服务器（补充）
- Web 缓存可暂存最近请求过的网页副本，当有新请求时直接返回缓存内容，降低响应时间，减少网络带宽消耗。  
- 常借助条件 GET 方法验证缓存是否有效（If-Modified-Since、If-None-Match 等）。
