# 第五章：传输层

## 一、传输层概述

### 1. 传输层功能
- 实现**进程到进程（端到端）的通信**
- 端到端可靠性覆盖整条路径的通信效果（路径包含中间路由器），但可靠性机制由端系统实现，不依赖路由器逐跳保证。
- **复用**：发送端多个进程共用同一个传输层协议发送数据
- **分用**：接收端传输层将收到的数据正确交付到不同的进程
- **差错检测**  
  - TCP 检测出错后丢弃报文段，并通知发送方重传  
  - UDP 检测出错后直接丢弃，且不通知发送方

### 2. 端口号（Port Number）
端口用来唯一标识主机内的特定进程。不同主机的端口号互相独立；同一主机上，TCP 和 UDP 的端口号空间也互相独立。

**套接字**（Socket = IP地址 : 端口号）唯一标识网络中一台主机的一个应用进程。

**五元组**：`(源 IP, 源端口, 目的 IP, 目的端口, 协议号)`，常用于标识一条传输连接/流。

**端口号分类**：
- 服务器使用的端口号  
  - 熟知端口号：0 ~ 1023，分配给最重要、熟知的应用  
  - 登记端口号：1024 ~ 49151  
- 客户端使用的端口号：**短暂端口号**（49152 ~ 65535）

---

## 二、传输层提供的两种服务

传输层向上层提供两种不同特性的传输协议：
- **TCP**：面向连接的、可靠的端到端传输服务
- **UDP**：无连接的、不可靠的端到端传输服务

---

## 三、UDP 协议

### 1. TCP 与 UDP 特点对比
| 特性   | UDP                | TCP             |
| ---- | ------------------ | --------------- |
| 首部长度 | 仅 8 B              | 20 ~ 60 B       |
| 报文分段 | 面向报文，不支持自动拆分与重装    | 面向字节流，支持自动拆分与重装 |
| 可靠性  | 无连接、不可靠（交给应用层处理）   | 面向连接、可靠         |
| 拥塞控制 | 不支持                | 支持              |
| 通信方式 | 一对一（单播）、一对多（广播/多播） | 仅支持一对一          |

### 2. UDP 数据报格式
UDP 首部共 8 B，包含：
- **源端口号**（16 bit）
- **目的端口号**（16 bit）
- **UDP 长度**（16 bit）：UDP **数据报的总长度（首部 + 数据）**，单位为字节。最小值为 8（仅首部），最大值 65535  
- **UDP 检验和**（16 bit）  
  计算检验和前需要添加 **12 B 伪首部**（伪首部仅用于计算，不传送），其中：  
  - 源 IP 地址（4 B）  
  - 目的 IP 地址（4 B）  
  - 全零字段（1 B）  
  - **协议字段 = 17**（1 B）  
  - UDP 长度（2 B）
  - IPv4 下 UDP 检验和可选，字段为全 0 表示未计算，接收端不应丢弃。

⚠️ **UDP 是面向报文的**：每次传输一个完整的 UDP 数据报，若 IP 层分片，对端再重装；UDP 本身不关心分片与重装。

## 四、TCP 协议

### 1. TCP 段格式
TCP 段的首部包含固定部分（20 B）和可变部分（选项），总首部 **20 ~ 60 B**。

| 字段            | 长度     | 说明                                                        |
| ------------- | ------ | --------------------------------------------------------- |
| 源端口           | 16 bit | 发送进程端口号                                                   |
| 目的端口          | 16 bit | 接收进程端口号                                                   |
| **序号 (seq)**  | 32 bit | 本报文段数据部分**第一个字节**在字节流中的序号（初始序号不一定为 0）                     |
| **确认号 (ack)** | 32 bit | 期望收到对方**下一个**报文段的第一个数据字节序号；表示该确认号之前的所有字节均已正确收到            |
| **数据偏移**      | 4 bit  | 以 4 B 为单位，指明 TCP 首部长度（因此首部最长 60 B）                        |
| 保留            | 6 bit  | 全部置 0                                                     |
| **URG**       | 1 bit  | =1 时紧急指针有效，表明有紧急数据需优先发送                                   |
| **ACK**       | 1 bit  | =1 时确认号有效；**仅第一次握手（SYN 段）时 ACK = 0**，其余所有 TCP 报文段 ACK = 1 |
| **PSH**       | 1 bit  | =1 时希望接收方尽快把数据上交应用层（常用于交互式通信）                             |
| **RST**       | 1 bit  | =1 表示出现严重差错，必须释放连接，也可用于拒绝一个非法的报文段                         |
| **SYN**       | 1 bit  | =1 表示这是一个连接请求或连接接受报文（**仅三次握手的前两次 SYN = 1**）               |
| **FIN**       | 1 bit  | =1 表示发送方数据已发送完毕，要求释放连接（**仅四次挥手的第一、三次 FIN = 1**）           |
| **窗口 (rwnd)** | 16 bit | 接收窗口大小，从本报文段确认号字段的值算起，接收方还能接收多少字节（用于流量控制）                 |
| 检验和           | 16 bit | 计算时也需要 12 B 伪首部，**协议字段 = 6**                              |
| 紧急指针          | 16 bit | 当 URG = 1 时，指示紧急数据的末尾位置                                   |
| 选项            | 可变     | 例如 MSS（最大报文段长度），**连接建立时双方协商 MSS（例如 MSS = 1000 B）**        |
| 填充            | 可变     | 使 TCP 首部长度对齐到 4 B 的倍数                                     |

**补充说明**：TCP 头部的基本标志位为 6 个：**URG / ACK / PSH / RST / SYN / FIN**（各 1 bit）。
CWR 与 ECE 属于 RFC 3168 的 ECN 拥塞通知扩展字段，不属于传统的六大基本标志位。

> 常用缩写全称：  
> - **MSS** : Maximum Segment Size（最大报文段长度）  
> - **rwnd** : Receiver Window（接收窗口）  
> - **cwnd** : Congestion Window（拥塞窗口）  
> - **ssthresh** : slow start threshold（慢开始门限）  
> - **MSL** : Maximum Segment Lifetime（最长报文段寿命）

### 2. TCP 面向字节流
TCP 把应用层交下来的数据看成一系列无结构的字节流，**面向字节流**传输，与 UDP 的面向报文形成对比。  
一次 TCP 连接中可以连续发送多个报文段。

## 五、TCP 连接管理

### 1. 三次握手（建立连接）
| 步骤 | 发送方状态 | 接收方状态 | 标志位与序号 |
|------|------------|------------|--------------|
| 第一次握手 | CLOSED → **SYN-SENT** | LISTEN | SYN = 1, ACK = 0, seq = x, ack = 无效 |
| 第二次握手 | SYN-SENT | LISTEN → **SYN-RCVD** | SYN = 1, ACK = 1, seq = y, ack = x+1 |
| 第三次握手 | SYN-SENT → **ESTABLISHED** | SYN-RCVD | SYN = 0, ACK = 1, seq = x+1, ack = y+1 |
| 完成后 | ESTABLISHED | SYN-RCVD → **ESTABLISHED** | 连接建立，可发送数据 |

> **为什么需要三次握手而不是两次？** 核心目的是**防止已失效的连接请求报文突然到达服务端，导致资源浪费**。想象：客户端发的第一个 SYN 因网络滞留迟迟未到，客户端超时后重发第二个 SYN 并正常完成通信。之后第一个 SYN 终于到达服务端——如果只需两次握手，服务端会误以为是新连接并分配资源等待数据，但客户端根本不会理睬，造成服务端资源白白浪费。第三次握手让客户端有机会判断"这个确认是对应我当前请求的"，从而丢弃过期的旧连接请求。

### 2. 四次挥手（释放连接）
| 步骤 | 主动方状态 | 被动方状态 | 标志位与序号 |
|------|------------|------------|--------------|
| 第一次挥手 | ESTABLISHED → **FIN-WAIT-1** | ESTABLISHED | FIN = 1, seq = x, ack = y |
| 第二次挥手 | FIN-WAIT-1 | ESTABLISHED → **CLOSE-WAIT** | ACK = 1, seq = y, ack = x+1 |
| 第三次挥手 | FIN-WAIT-1 → **FIN-WAIT-2**（收到 ACK 后） | CLOSE-WAIT → **LAST-ACK** | FIN = 1, ACK = 1, seq = y, ack = x+1 |
| 第四次挥手 | FIN-WAIT-2 → **TIME-WAIT** | LAST-ACK | ACK = 1, seq = x+1, ack = y+1 |
| 最终 | TIME-WAIT（等待 2MSL）→ CLOSED | LAST-ACK → **CLOSED** | — |

- 主动关闭方在发送最后一次 ACK 后进入 **TIME-WAIT** 状态，等待 **2MSL**（最长报文段寿命的两倍）后进入 CLOSED。
- **为什么要等 2MSL？** ① 确保最后一个 ACK 能到达被动关闭方：若丢失，被动方会重发 FIN，主动方需能在 TIME-WAIT 期间收到并重发 ACK；② 确保本连接的所有报文段从网络中消失，避免与后续新建连接的报文混淆（"迷途报文"问题）。
- CLOSE-WAIT 状态的持续时间取决于被动方应用层何时调用 close，不一定是极短阶段。

### 3. 同时打开与同时关闭

**同时打开（Simultaneous Open）**：
- 两端同时主动发送 SYN 报文（即双方几乎同时发起连接请求）。
- 状态变迁：双方均从 CLOSED → SYN-SENT → SYN-RCVD → ESTABLISHED。
- 双方各发一个 SYN，各收一个 SYN，并各回一个 SYN+ACK，总共交换 **4 个报文段**（而非正常三次握手的 3 个）。
- 最终双方都进入 ESTABLISHED 状态。

**同时关闭（Simultaneous Close）**：
- 两端同时主动发送 FIN 报文（即双方几乎同时调用 close）。
- 状态变迁：双方均从 ESTABLISHED → FIN-WAIT-1 → CLOSING → TIME-WAIT → CLOSED。
- 双方各发一个 FIN，各收一个 FIN，并各回一个 ACK，总共交换 **4 个报文段**（与正常四次挥手的报文数相同，但中间状态不同）。
- 双方都经过 CLOSING 状态（而非正常流程中的一方 CLOSE-WAIT、另一方 FIN-WAIT-2），最终都进入 TIME-WAIT 等待 2MSL 后关闭。

## 六、TCP 可靠传输

### 1. 确认机制
- **立即确认**：每收到一个 TCP 报文段，就立即返回一个 ACK 段（即使收到的是失序报文段也要立即返回确认号相同的 ACK）。  
- **推迟确认**：TCP 标准规定推迟确认的时间最多不超过 **0.5 秒**；  
  - 如果自己也有数据要发送，可以**捎带确认**（ACK 段携带数据）。  
  - 若连续收到两个 MSS 大小的报文段，应立即返回 ACK 段。  

两种 ACK 段形态：
- **专门确认**：仅 TCP 首部，不携带数据。
- **捎带确认**：确认的同时顺带发送数据。

### 2. 超时重传
- 每发出一个报文段，就启动一个重传计时器。若计时器到期仍未收到确认，则重传该报文段，并重置计时器。

### 3. 快重传（Fast Retransmit）
- 当发送方**连续收到三个相同的确认号**（即收到三个重复的 ACK，不包括最初的正常 ACK），立即重传该确认号对应的报文段。  
- 这三个重复 ACK 说明失序，对应的报文段已丢失，不必等待超时，从而提高效率。

## 七、TCP 流量控制

流量控制是**端到端**的，利用接收窗口 rwnd 限制发送方速率，防止接收方缓冲区溢出。
- 从第二次握手开始，接收方在 TCP 报文段的窗口字段中通告自己的接收窗口大小 rwnd。  
- **发送窗口上限 = min(rwnd, cwnd)**，其中 cwnd 为拥塞窗口（见拥塞控制）。
- 通过动态改变 rwnd 的大小进行流量控制。

**Nagle 算法与 Clark 算法**

| 算法 | 解决的问题 | 核心思想 |
|:---|:---|:---|
| **Nagle 算法** | 减少网络中小分组的数量 | 发送方在收到前一个报文段的 ACK 之前，先缓冲后续的小数据；当积累到一个 MSS 大小或收到 ACK 时才发送。减少大量小包浪费带宽。 |
| **Clark 算法** | 解决糊涂窗口综合征（Silly Window Syndrome） | 接收方在窗口太小时不通知发送方，而是等到窗口至少能容纳一个 MSS 或缓冲区空间的一半时再通告，防止发送方发送极小的报文段。 |

> **辨析：发送窗口 vs 还能继续发送**
> - **发送窗口** `= min(rwnd, cwnd)`，这是允许发送的最大字节数的理论上限，仅受拥塞窗口和接收窗口的公式制约，不考虑接收方是否已确认。
> - **还能继续发送的数据量** `= 发送窗口 − 已发送但未确认的数据`，即在发送窗口的基础上，还需减去仍在网络中等待确认（in-flight）的字节数。
> - 简言之：发送窗口决定"最多能发多少"，"还能继续发"决定"现在立刻还能发多少"。

## 八、TCP 拥塞控制

### 1. 流量控制 vs 拥塞控制
- **流量控制**：局部、端到端，控制发送方发送速率匹配接收方能力。  
- **拥塞控制**：全局，抑制整个网络中的总数据量，降低路由器负载，防止网络拥塞。

**判断网络拥塞的依据**：
- 每个报文段都能及时收到 ACK → 未拥塞  
- 发生超时重传 → **严重拥塞**  
- 收到重复 ACK（冗余 ACK）→ **轻微/可能出现拥塞**  

发送窗口上限值 **= min(rwnd, cwnd)**，cwnd 以 MSS 为单位进行增减。

### 2. 慢开始（Slow Start）与拥塞避免（Congestion Avoidance）
- **慢开始门限 ssthresh** 初始可设较大值（如教材中典型初始值为 16 个 MSS，但实际由系统设置）。  
- **慢开始算法**（cwnd < ssthresh）：  
  每收到一个对新报文段的**确认**，cwnd 就增加一个 MSS（按指数增长；如果每 RTT 收到多个 ACK，cwnd 在一个 RTT 内可能成倍增加）。  
- **拥塞避免算法**（cwnd ≥ ssthresh）：  
  **每经过一个往返时间 RTT**，cwnd 增加一个 MSS（线性增长）。不能简单写成“每收到一个 ACK 加一”，那样会使增长过快。

**发生超时重传时的处理**：
- ssthresh = max(cwnd / 2, 2)（门限至少 2 MSS）  
- cwnd = 1 MSS  
- 重新执行慢开始算法

### 3. 快重传与快恢复（Fast Retransmit and Fast Recovery）
- **快重传**：收到三个重复 ACK 时，立即重传丢失的报文段，不等超时。  
- **快恢复**：  
  - ssthresh = cwnd / 2（且 ssthresh ≥ 2）  
  - cwnd = ssthresh（或将 cwnd 设为 ssthresh + 3×MSS，具体实现略有不同），然后直接进入**拥塞避免**阶段（不执行慢开始），从而使窗口呈线性增长，快速恢复已减半的传输速率。

> 传统处理（超时）会让 cwnd 骤降至 1，快恢复则将 cwnd 降为一半，避免传输速率剧烈下降。

### 4. AIMD（加性增乘性减）

**AIMD（Additive Increase Multiplicative Decrease）** 是 TCP 拥塞避免阶段的核心策略，也是整个拥塞控制的设计精髓。

- **加性增（AI）**：在拥塞避免阶段，每经过一个 RTT，cwnd 增加一个 MSS，窗口线性增长——缓慢试探可用带宽的上限。
- **乘性减（MD）**：一旦检测到丢包（超时或收到冗余 ACK），立即将 cwnd 减半，快速释放网络资源以缓解拥塞。
- 核心思想："谨慎探测、快速退让"——增长时保守，遇拥塞时果断降速。慢开始+拥塞避免（超时）或快重传+快恢复（冗余 ACK）均为 AIMD 的具体实现形式。
